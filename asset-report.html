The application uses three external storage systems plus a PostgreSQL database for metadata. Every file uploaded through the UI is tracked in the database with its path, URL, size, and who uploaded it.

📌 Key Concept: Dual Storage

Renderings, project covers, and quote PDFs are stored in both Blob (for fast UI display) and Dropbox (for archive). The system always prefers the Blob URL when serving to users. If Blob isn't configured or fails, it falls back to a Dropbox temporary link.

Vercel Blob Storage — CDN Assets

Vercel Blob is used for assets that need instant loading in the UI. These are publicly accessible via CDN URLs.

What Blob Path Also Mirrored to Dropbox?
User Avatars avatars/{userId}/ No
Project Covers project-covers/{projectId}/ Yes
Spec Item Images spec-items/{projectId}/ No
Quote Documents quote-documents/{projectId}/ Yes
Renderings orgs/{orgId}/projects/{projectId}/rooms/{roomId}/ Yes
⚠️ Storage Limit Awareness

Vercel Blob has storage limits: 1 GB on Hobby, 5 GB on Pro. An admin Blob Status page exists to monitor usage broken down by file type and age. A manual Blob-to-Dropbox migration route can move old files out of Blob to free space.

How Blob URLs Look

Public URLs follow this pattern: https://{hash}.public.blob.vercel-storage.com/{pathname}

Spec Book Generation

Spec Books are auto-generated PDF documents that combine renderings, drawings, and CAD files into a single professional document.

Generation Process
  1. User selects which sections to include (Floorplans, Lighting, Electrical, Plumbing, etc.)
  2. User selects which rooms to include
  3. System fills in the cover page (project name, client, date)
  4. For each section: pulls CAD files from DropboxFileLink, converts to PDF if needed (via CloudConvert)
  5. For each room: pulls approved renderings (from Blob or Dropbox URLs)
  6. Generates the combined PDF using the PDF generation service
  7. Uploads final PDF to Dropbox at 11- SOFTWARE UPLOADS/Spec Books/Generated/

Spec book versions are tracked with version numbers (1.0, 1.1, 1.2), and include metadata like page count, file size, download count, and who generated it.

How Files Are Served Back to Users

When displaying files in the UI, the app uses a priority chain to pick the best URL:

URL Resolution Priority
  1. Blob URL — If stored in Vercel Blob → use directly (fastest, CDN-backed)
  2. Dropbox Temporary Link — Generated on-demand, expires in 4 hours
  3. Dropbox Shared Link — Permanent, uses ?raw=1 for direct embedding
  4. Stored URL — Whatever URL is saved in the database (fallback)

PDF Thumbnails: Generated server-side using unpdf + @napi-rs/canvas and cached for 1 hour. This avoids loading full PDFs just to show a preview.

Watermarking

Renderings are watermarked with the company logo (Meisner Interiors) at upload time:

  • Logo placed at bottom-left corner with 30px padding
  • Logo sized to 8% of image height
  • Opacity: 85% (slightly transparent)
  • If watermarking fails, the original image is used silently

Critical Issues

🗑️

Orphaned Blob Files Never Cleaned Up CRITICAL

When a rendering or asset is deleted, the Blob URL stored in metadata is never deleted from Vercel Blob storage. Files accumulate indefinitely, consuming storage quota. Only a manual admin cleanup exists, and it doesn't detect orphans.

📂

Project Deletion Leaves Cloud Files Behind CRITICAL

When a project is deleted, database records are cleaned up but Dropbox files and Blob files are NOT removed for most asset types. Only rendering-specific Dropbox folders get explicit cleanup. All other files (drawings, procurement, source files, photos) remain as orphans.

🔄

No Retry Logic for Dropbox API Calls CRITICAL

All Dropbox operations (upload, delete, folder creation) have zero retry logic. A single transient network glitch = permanent failure. The user sees an error and must retry manually. For batch operations, partial failures can leave things in an inconsistent state.

📧

No Email Attachment Size Validation CRITICAL

When sending transmittals via email, there is no validation of total email size. Users can attach many large PDFs that exceed the 25 MB email provider limit, causing silent delivery failures. The user sees "Sent" but the email never arrives.

📄

Drawing Deletion is Soft-Delete Only CRITICAL

When drawings are deleted, they are only set to ARCHIVED status in the database. The actual PDF files in Dropbox are never removed. Over time, this means Dropbox fills up with files that no longer appear in the UI but still consume storage.

💥

Partial Upload Failure = Orphaned Files CRITICAL

When uploading multiple renderings, if one file fails mid-way, previously uploaded files are already persisted in Blob/Dropbox. The API returns an error, so the user may retry, creating duplicates of the successful files while only the failed one gets re-uploaded.

🔍

No File Search Across Project CRITICAL

Users cannot search for files by name, type, or content. Finding a specific drawing or rendering among hundreds of files requires manual browsing through folders. No full-text search exists for file names or metadata.

High Severity Issues

🤫

Silent Blob Fallback HIGH

If Blob upload fails, the system silently falls back to Dropbox-only storage. The user is never informed that their file isn't CDN-backed, which may cause slower load times.

🖼️

Watermark Failure Silently Ignored HIGH

If the watermark fails to apply to a rendering, the original unwatermarked image is saved and sent to clients. There's no notification that the company logo wasn't applied.

🔒

Inconsistent Permission Checks on File Delete HIGH

Asset deletion only checks org-level access, not project-level or role-based access. A user in the same org could theoretically delete files from projects they shouldn't have access to. Only project deletion checks OWNER role.

📏

Inconsistent File Size Limits HIGH

Different upload endpoints have different limits: 10 MB for renderings, 25 MB for procurement, 50 MB for spec book PDFs, 100 MB for videos. The Send Files dialog has no visible size validation at all.

🔐

No File Content Validation HIGH

File type validation relies on MIME type (from the browser) or extension, but never validates actual file content. A malicious file could be renamed to .pdf with a spoofed MIME type and pass validation.

📊

No Download/Access Logging HIGH

Activity logs track uploads but not downloads. There's no audit trail showing who accessed, viewed, or downloaded which files and when.

♻️

No Content-Based Deduplication HIGH

Users can upload the exact same file multiple times. No hash-based checking exists. The only deduplication is for drawing titles/numbers in the Send Files dialog. Storage waste accumulates over time.

⏸️

No Resume on Upload Failure HIGH

If an upload is interrupted (network drop, browser refresh), there is no resume capability. Large files must be re-uploaded entirely. No localStorage persistence of pending uploads.

⚡

No Dropbox API Rate Limiting HIGH

No rate limiting is implemented for Dropbox API calls. Bulk operations (creating project folders, uploading many files) could hit Dropbox rate limits and cause cascading failures.

Missing Features & Recommendations

Top Priority Recommendations

1

Implement Blob File Cleanup Cron Job

Create a scheduled daily job that queries all Asset records, collects referenced Blob URLs, then deletes any Blob files not in that set. Also delete Blob files when assets are deleted via the API. This prevents unbounded storage growth.

2

Add Email Size Validation Before Send

Before sending a transmittal email, calculate total attachment size. If over 20 MB, either (a) warn the user and suggest fewer files, (b) upload to Dropbox and send a download link instead, or (c) split into multiple emails.

3

Add Retry Logic to Dropbox Operations

Implement exponential backoff retry (3 attempts with 1s, 2s, 4s delays) for all Dropbox API calls. This handles transient network issues gracefully and prevents permanent failures from temporary glitches.

4

Implement Transactional File Deletion

When deleting an asset, wrap the operation so that all storage locations (DB + Blob + Dropbox) are cleaned up together. If any deletion fails, log it and queue for retry rather than leaving orphaned files.

5

Add File Search to Project Files

Add a search bar to the All Files tab that searches across file names, drawing titles, sections, and metadata. This is a high-impact UX improvement for projects with many files.
